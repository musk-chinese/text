### 1）基础定义部分（typedef、全局 byte 数组、`HOST` 结构体）
- 功能：统一定义“字节”类型、预先放好发送/接收双方的端口、IP、MAC 地址，方便后面填快递单使用。
- 关键代码片段（来自两个文件）：
  - `typedef unsigned char byte;`
    - C 语法解释：把 `unsigned char` 取个短名叫 `byte`。就像给“邮包单位”取个便于书写的别名。
    - 为寄快递做准备：定义了包装盒里的最小颗粒（一个字节），后面所有地址和字段都用它来表示。
  - 全局数组，例如：
    - `byte srcPort[2] = {0x00,0xee};`
    - `byte srcIP[4] = {0x00,0x01,0x02,0xee};`
    - `byte srcAddress[6] = {0x00,0x11,0x22,0x33,0x44,0xee};`
    - 解释：这些是固定的源（sender）端口、IP、MAC 地址，每个元素是一个字节。比如 IP 有 4 个字节，MAC 有 6 个字节。
    - 类比：这些就是寄件人（源主机）的姓名/门牌/单位，提前写在寄件单上，便于装箱时贴上。
  - `struct HOST { byte port[2]; byte ip[4]; byte address[6]; } srcHost, desHost;`
    - C 语法解释：`struct` 是“结构体”，你可以把它想成一个“打包箱”或“寄递单格子”，里面能同时装下不同种类的信息（端口、IP、MAC）。`srcHost`、`desHost` 是两个这样的箱子（源和目的）。
    - 类比：`HOST` 就像一个表格，里边写着“寄件人电话/地址/身份证号”，填好后便于交给邮局。
  - `void initHots() { memcpy(srcHost.port, srcPort, 2); ... }`
    - `memcpy` 语法解释：将一块内存的数据复制到另一块内存。类似把预先写好的表格内容直接套到寄件单上。
    - 这是在为寄快递做准备：把事先写好的地址块粘贴到寄件单的固定字段上。

注意：每个全局数组（`srcPort`、`desIP` 等）都相当于写好的一行寄递信息，封装时会直接把它们拷贝到不同层的头部里。

### 2）各层协议结构体（`OCP`、`UDP`、`IP`、`MAC`）
这些结构体代表不同的“包装层”，从内到外分别是应用层（OCP）、传输层（UDP）、网络层（IP）、链路层（MAC / 以太网帧）。

总体类比：把一件东西（消息）装进小盒子（OCP），再把小盒子放进中盒（UDP），再放进大盒（IP），最后贴上外包装和邮政标签（MAC），每个盒子外面都有标签（字段）说明是谁寄的、寄给谁、尺寸、校验等。

- `struct OCP`（应用层）
  - 字段（逐一解释）：
    - `byte mark1[2];`、`byte mark2[2];`：两道标志位，像快递单上的“开始”和“结束”标记，帮助接收方识别包的边界。
    - `byte protocol[2];`：协议版本号，类似“快递单样式编号”。
    - `byte control[2];`：控制字段，表示操作类型（例如发送消息）。
    - `byte srcID[8];`、`byte desID[8];`：源/目的用户 ID（更上层的标识）。
    - `byte sum[2];`：OCP 层的校验和（像里面一张小单据的签名）。
    - `byte reservation[6];`：预留字段（暂为空，可以以后扩展）。
    - `byte len[2];`：这整个 OCP 的长度（头 + 数据），用两个字节表示。
    - `byte data[256];`（在 `client.cpp` 使用数组）或 `byte* data;`（`server.cpp` 使用指针）：
      - 这里是实际的消息数据（小盒子里的东西）。
  - C 语法解释（结构体 + 数组/指针）：
    - 数组 `byte data[256]` 在结构体中是“固定大小的空间”，而 `byte* data` 是“指向别处内存的地址”，用指针需要保证它指向的内存在有效期内（这是 `server`/`client` 里设计不同的地方，后面会讲风险）。
  - 这是在填快递单并把东西放进小盒子：在 OCP 里写好寄件人 ID、收件人 ID、长度、并把消息放进去。

- `struct UDP`（传输层）
  - 字段：
    - `byte srcPort[2]; byte desPort[2];`：源/目的端口，相当于应用程序的“房间号”或“部门号”。
    - `byte sum[2];`：UDP 层的校验和。
    - `byte len[2];`：UDP 包的长度（UDP 头 + payload）。
    - `OCP ocp;`：把上面的 OCP 放到 UDP 的 payload 中。
  - 类比：这是把小盒子放进中盒子，并贴上中盒的“发/收部门号”标签。

- `struct IP`（网络层）
  - 字段：
    - `byte header[1];`：版本 + 头部长度（例如 `0x45` 表示 IPv4, 头部长度 5 个 4 字节 = 20 字节）。
    - `byte service[1];`：服务类别（优先级等）。
    - `byte len[2];`：IP 包的总长度（IP 头 + UDP + OCP）。
    - `byte layer2[4];`：标识、标志、片偏移（分片相关）。
    - `byte time[1];`：生存时间 TTL（包可以在网络上跳多少次）。
    - `byte protocal[1];`：上层协议（这里 `0x11` 表示 UDP）。
    - `byte sum[2];`：IP 头部检验和（IP 层做的签名）。
    - `byte srcIP[4]` / `desIP[4]`：源 / 目的 IP 地址（网络位置）。
    - `UDP udp;`：包里面封装的 UDP。
  - 类比：IP 就像邮政的“国内/国际运单”，写上目的地城市（IP）与邮递要求，外面还有计费和路由信息。

- `struct MAC`（链路层 / 以太网帧）
  - 字段：
    - `byte srcAddress[6]; byte desAddress[6];`：源/目标 MAC（物理网卡地址）。
    - `byte type[2];`：指示载荷类型（比如 IPv4）。
    - `IP ip;`：封装的 IP 数据报。
    - `byte FCS[4];`：帧校验（FCS，通常是 CRC32），用于检测帧在传输中是否被损坏。
  - 类比：MAC 是外包装，贴上“发件人门牌”和“收件人门牌”，并在包装角落贴上防篡改条（FCS）。

关键点：每一层结构体里基本都是“上层东西 + 本层头部字段（长度、校验、地址）”。

### 3）工具函数（`crc32`、`shortToByte`/`bytesToShort`、`displayByte`、`IPCheckSum`、`UdpCheckSum`）
这些函数都是“工具箱”，负责转换、打印、计算校验等。

- `crc32_for_byte` / `crc32` （计算 CRC32，用作 FCS）
  - 干嘛的：CRC32 是一种检测数据在传输中有没有被损坏的算法（将整段字节通过算法压成 4 字节的表示）。
  - 网络类比：像给包裹贴的“封条条形码”，收件人重新计算一遍条形码能检测包装是否被篡改或损坏。
  - 关键语法点（举例）：
    - `r = (r & 1 ? 0 : (unsigned int)0xEDB88320L) ^ r >> 1;`
      - 解释：这行用到了按位与 `&`、条件运算 `?:`、按位异或 `^`、右移 `>>`。把它大白话化：
        - `r & 1`：检查 `r` 的最低位是不是 1，像给数字做按位体检；
        - 条件 `? :`：如果最低位是 1，就用 0，否则用一个常数（多项式常数）；然后把 `r` 右移 1 位（像把积木整体往右移一格），并把结果做 XOR（异或，相当于两组积木做特殊混合）。
      - 这是在“给包裹做一圈按位体检并更新 CRC 临时值”。
    - `return crc;` 最后返回一个 32-bit 的 CRC 值。
  - 注意：实现细节里有移位和按位运算，都可以按“积木左移/右移”和“按位体检”来理解。

- `shortToByte(short i, byte* bytes)` 与 `bytesToShort(byte* bytes)`
  - 干嘛的：把一个 16-bit 的数字拆成两个字节（或把两个字节合成一个 16-bit 数）。这是“端序转换”。
  - 语法/行解释：
    - `bytes[0] = (byte) ((0xff00 & i) >> 8);`
      - `0xff00 & i`：把 `i` 的高 8 位挑出来（按位与），就像用筛子抓出高一层积木。
      - `>> 8`：把这些高 8 位右移到低位位置（像把积木缩到最小单位）。
      - 结果放到 `bytes[0]`，这表明使用的是“大端序”（高位在前）。
    - `short num = bytes[1] | (bytes[0] << 8);`
      - `bytes[0] << 8`：把第一个字节左移 8 位（像把它放回高位），然后用按位或 `|` 合并两个字节成一个短整型。
  - 类比：把一个两位数字拆成“十位”和“个位”，或把十位和个位组合成一个完整数值。

- `displayByte(const char str[], byte* arr, int len)`
  - 干嘛的：把一段字节以十六进制打印出来，便于调试。
  - 语法点：`for` 循环遍历数组，`printf("%02x ", arr[i]);` 把每个字节打印为两位 16 进制。
  - 类比：拿出寄递单，用放大镜把每一行字段逐字打印出来给人看。

- `IPCheckSum(byte *ip_head_buffer, short ip_hdr_len)`
  - 干嘛的：计算 IP 头部的校验和（把 IP 头的若干 16-bit 单元累加并做进位折叠，最后取反）。
  - 网络概念（大白话）：就像把运单上每一项金额相加，得到一个总和，然后把进位折叠回去，再做取反，目的是生成一个签名。接收端同样计算签名并比较，若不一致说明运单被修改。
  - 关键行解释：
    - 循环里：
      - `byte t1 = *ip_head_buffer++; byte t2 = *ip_head_buffer++; short s = (short)(t1<<8)|t2;`
        - 逐字节读取两个字节，`t1<<8` 把第一个字节左移 8 位（像把高位搬上去），然后用 `|` 把两者合成一个 16-bit 数。
      - `check_sum += s;` 将 16-bit 单元累加到校验和里。
    - 进位折叠：
      - `check_sum = (check_sum & 0x0000FFFF) + (check_sum >> 16);`
        - 这是把超过 16-bit 的高位“摘下来”加回低位，类似“把超出的位数当零钱换回”。
      - `check_sum = ~check_sum;` 最后取反（按位 NOT），作为校验和的最终值。
  - 类比：算账并签名，接收时重新算账比较签名。

- `UdpCheckSum(byte* ip_src_addr, byte* ip_dst_addr, byte *udp_buffer, short udp_size)`
  - 干嘛的：UDP 校验和的计算使用了一个“伪首部”（包含 IP 源/目的地址、协议号和 UDP 长度）再加上 UDP 的内容一起计算，这样可以防止 IP 或端口被误改。
  - 关键点：先构造 `rawBuffer`，把伪首部放前面，然后把 UDP 数据拷贝到后面，用 `IPCheckSum` 对整块数据求和。
  - 类比：为了更可靠地验签，不光检查包装单（UDP），还把地址信息也一起算进去，好像在包外再贴上一层“邮政路线信息”，方便校验。

这是在为寄快递做准备：这些工具是在称重、计算费用、贴防伪标的时候会用到的工具。

### 4）封装函数（`encapsulateOCP` / `encapsulateUDP` / `encapsulateIP` / `encapsulateMAC` / `encapsulateData` / `macToByte`）
这是“装箱”的过程，把原始消息每一层逐渐包起来，并生成最终可以传输的字节流（frame）。

- `encapsulateOCP(byte data[], int len)`
  - 干嘛的：把原始字节 `data` 放进 OCP 结构体，填写标志、协议、控制、源/目的 ID、预留字段，设置 `len`（长度）。
  - 关键代码：
    - `OCP ocp = {0};` —— 把结构体内存清零（相当于拿个空表格准备填写）。
    - `memcpy(ocp.mark1, mark1OCP, 2);` —— 把标志字段拷贝进 OCP 的 `mark1`（把预先写好的“开始标识”贴到单子上）。
    - `shortToByte(lenOCP, ocp.len);` —— 把计算出的 OCP 总长度写到 `len` 字段（把箱子尺寸写在单子上）。
    - `memcpy(ocp.data, data, len);` —— 把实际消息拷贝进 `ocp.data`（把东西放进小盒子）。
  - 类比：把内件放进小盒子，贴好小盒子单（标志、种类、发/收信息、尺寸）。
  - 注意：`server.cpp` 原始版本用了指针 `ocp.data = data`（指向外部数组），而 `client.cpp` 使用固定数组 `data[256]`。指针更节省空间但要小心内存有效期（如果原数组被释放或超出作用域就有问题）。

- `encapsulateUDP(OCP &ocp)`
  - 干嘛的：在 UDP 头部写入源/目标端口、初始校验和，然后把 OCP 放到 UDP，计算并写入 UDP 长度。
  - 关键行：
    - `memcpy(udp.srcPort, srcPort, 2);` 把全局源端口写到 UDP 头。
    - `short lenUDP = bytesToShort(ocp.len) + 8;` UDP 总长度 = OCP 长度 + UDP 头（8 字节）。
    - `shortToByte(lenUDP, udp.len);` 把长度写入 UDP 的 `len` 字段。
  - 类比：把小盒子放进中盒，贴上“部门编号”和“中盒尺寸”。

- `encapsulateIP(UDP &udp)`
  - 干嘛的：安装 IP 头部（版本、服务、源/目的 IP、协议），并把 UDP 放入 IP，计算 IP 总长度（IP 头 20 字节 + UDP）。
  - 关键行：
    - `short lenIP = bytesToShort(udp.len) + 20;` 计算 IP 总长度。
    - `shortToByte(lenIP, ip.len);` 写回 IP 的 `len` 字段。
  - 类比：把中盒放进大盒，写上目的城市、邮编等信息。

- `encapsulateMAC(IP &ip)` 和 `encapsulateData(byte data[], int len)`
  - `encapsulateMAC`：把 IP 放进 MAC，填 MAC 的源/目的地址与类型字段（以太网类型）。
  - `encapsulateData`：一步调用前面的封装函数，把原始数据封成最终的 MAC 结构体。

- `macToByte(MAC& mac, short lenMAC, byte frame[])`
  - 这是把结构体形式的 MAC（含 IP/UDP/OCP）转换成一个连续的字节数组（真正在网线/文件里要写的字节序列）。
  - 干嘛具体做的（按偏移写入）：
    - `memcpy(&frame[0], mac.srcAddress, 6);` 把 MAC 源地址放在帧的起始位置（0-5）。
    - `memcpy(&frame[6], mac.desAddress, 6);` 把目的 MAC 填写到 6-11。
    - `memcpy(&frame[12], mac.type, 2);` 类型字段放在 12-13。
    - `memcpy(&frame[14], ip.header, 1);` IP 头从偏移 14 开始写（14 是以太网头长 14 字节的原因）。
    - 后续依次把 IP、UDP、OCP 的各个字段放到固定偏移（例如 UDP 从 34 开始，OCP 从 42 开始，数据从 76 开始），这些偏移是基于每层头部的固定长度算出来的（14 + 20 = 34，34 + 8 = 42，42 + 34 = 76）。
  - 注意并计算校验：
    - `short sumIP = IPCheckSum(&frame[14],20); shortToByte(sumIP, ip.sum); memcpy(&frame[24], ip.sum,2);`
      - 先把 IP 头部分的字节放好，然后计算 IP 校验和，再把结果写回 IP 头的 `sum` 字段（更新帧里的内容）。
    - `short sumUDP = UdpCheckSum(ip.srcIP,ip.desIP,&frame[34],udp_len); shortToByte(sumUDP,udp.sum); memcpy(&frame[38],udp.sum,2);`
      - 计算 UDP 校验和并写入 UDP 头。
    - `short sumOCP = IPCheckSum(&frame[42],32); shortToByte(sumOCP,ocp.sum); memcpy(&frame[64],ocp.sum,2);`
      - 用 IPCheckSum 函数计算 OCP 的“头部校验和”。
    - 最后 `unsigned int FCS = crc32(frame, lenMAC - 4); memcpy(&frame[lenMAC - 4], &FCS, 4);`
      - 计算整帧的 CRC（不包括最后 4 字节位置），把 CRC 写到帧尾（FCS 字段）。
  - 逐字节写入的意义：网络上传输的就是这些按顺序排列的字节，接收端会按相同偏移解析。
  - 类比：把盒子按顺序装好，贴标签 → 然后再去称重并在单子上写重量信息、验证码 → 最后贴上封条（FCS）并投入邮筒。

这是在填快递单并把包裹贴好封条准备送出。

### 5）解封装函数（`bytesToMAC`、`decapsulateMAC`、`checkSum`、`checkFCS`）
这是“拆包”和“验货”的过程。

- `bytesToMAC(byte* frame, byte Data[])`
  - 干嘛的：把连续的字节帧（`frame`）按偏移解析回 `MAC` / `IP` / `UDP` / `OCP` 这些结构体字段，并把 OCP 的数据拷贝到 `Data`（外部缓冲区）。
  - 关键行解释：
    - `memcpy(mac.srcAddress, &frame[0], 6);` 把前 6 字节读为源 MAC 地址（把包装上的标签抄下来放在表格里）。
    - `memcpy(ip.header, &frame[14], 1); memcpy(ip.len, &frame[16], 2);` 把 IP 头对应位置读出来。
    - 计算 `lenMAC = bytesToShort(ip.len) + 18;`：从 IP 的 `len` 得到 IP 总长度，然后加上 MAC/以太网头尾部长度得到整帧长度（注意代码用的是 `+18`，它把以太网头和 FCS 等综合在一起）。
    - `int ocp_data_len = lenMAC - 80; memcpy(Data, &frame[76], ocp_data_len); memcpy(ocp.data, &frame[76], ocp_data_len);`
      - 把 OCP 的数据部分（从偏移 76 开始）复制出来，既放到临时 `Data` 缓冲，也放回 `ocp.data`（结构体内固定数组）。
  - 类比：拆开外包装，取出里面的中盒、小盒，把单据上每一项抄下来，最后把内件摆出来给验货。

- `decapsulateMAC(MAC &mac)`
  - 干嘛的：检查目的地址是否匹配本机（`desHost`），再检查 IP / UDP / OCP 的目的字段确保确实寄给了我。如果都通过，说明消息“收到了”。
  - 关键代码：
    - `if(!check(desHost.address, 6, mac.desAddress, 6))` —— 检查目的 MAC 地址是否匹配本机的 `desHost.address`（接收端的地址）。
    - `else if(!check(desHost.ip,4,ip.desIP,4))` —— 检查 IP 地址是否匹配。
    - `else if(!check(desHost.port,2,udp.desPort,2))` —— 检查端口。
    - `else { printf("Message received successfully.\n"); }`
  - 类比：验收员核对门牌（MAC）、城市（IP）、部门（端口）是否是他。如果都对，就把内件交给部门。

- `checkFCS(byte* frame, short lenMAC)`
  - 干嘛的：计算帧内容（除了尾部 4 字节）CRC，与帧尾的 FCS（最后 4 字节）做比较，判断传输中是否损坏。
  - 关键行：
    - `unsigned int FCS_int = crc32(frame, lenMAC - 4);` —— 重新计算 CRC。
    - `memcpy(FCS_frame, &frame[lenMAC - 4], 4);` —— 取出帧中原来的 FCS。
    - `if (check(FCS_cp,4,FCS_frame,4)){ printf("FCS check passed.\n"); }`
  - 类比：收件人把条形码重新扫一遍，看是否与包上贴的一致。

- `checkSum(byte* frame)`
  - 干嘛的：分别检查 IP 头的校验和、UDP 校验和、OCP 的头校验和是否正确。
  - 关键步骤：
    - `short num1 = IPCheckSum(&frame[14],20);` —— 计算 IP 头校验和。
    - `short num2 = UdpCheckSum(&frame[26],&frame[30],&frame[34],udp_len);` —— 计算 UDP 校验和，需要传入 IP 源/目的地址及 UDP 部分。
    - `short num3 = IPCheckSum(&frame[42],32);` —— 这里用 IPCheckSum 计算 OCP 的校验和（实现上把 OCP 头部当 32 字节来算）。
  - 类比：对不同级别的单据都做签名验证，确保单据本身没有被人篡改。

这是在邮局拆包验货：一层层核对标签和封条，确保既是给我、也没被损坏。

### 6）文件操作函数（`writeMessage`、`readMessage`、`getLenMAC`）
这些函数把帧写入或读出 `test.txt`，模拟把帧放到“共享媒介/邮筒”里。

- `writeMessage(byte* data, short len, FILE *fp)`
  - 干嘛的：把内存中的 `data` 写入打开的文件 `fp`，写入 `len` 个字节。
  - 关键行：
    - `size_t write_len = fwrite(data, sizeof(byte), len, fp);`
      - `fwrite` 是“把内存块写到文件”的 C 标准库函数，参数是源地址、每个单元大小、单元数量、文件指针。
    - `fclose` 在函数外统一由调用方管理（代码里已改为不在函数内部关闭），避免重复关闭。
  - 类比：把包裹投入邮筒（写文件）并让主流程决定什么时候关门（何时关闭文件句柄）。

- `readMessage(byte* data, short len, FILE *fp)`
  - 干嘛的：从文件 `fp` 中读取 `len` 个字节保存到 `data`。
  - 关键行：
    - `size_t read_len = fread(data, sizeof(byte), len, fp);`
      - `fread` 的行为类似 `fwrite`，只是读的方向相反。
  - 类比：邮递员从邮筒取出包裹并把字节逐个搬到手推车上。

- `getLenMAC(FILE *fp)`
  - 干嘛：从文件中先读取前 18 字节（包含 IP 的 len 字段）来判断总帧长度，然后把文件指针复位（`fseek(fp, 0, SEEK_SET)`）。
  - 关键行：
    - `byte data[18] = {0}; size_t read_len = fread(data, sizeof(byte), 18, fp);`
    - `byte IPLEN[2]; memcpy(IPLEN, &data[16], 2); short lenIP = bytesToShort(IPLEN); return (lenIP > 0) ? (lenIP + 18) : 0;`
      - 解释：IP 的 `len` 字段在偏移 `16` 位置（因为 14 字节以太网头 + 2 字节 header/service），`lenIP` 是 IP 报文长度；加上外层的 18 字节（以太网头 + FCS）得到 MAC 长度。
  - 类比：打开快递箱的一角查看快递单上写的尺寸（IP length），从而估算整箱尺寸（MAC frame length），然后再把箱子关上准备读完整个箱子。

### 7）两个 `main` 函数（发送端主逻辑、接收端主逻辑）

先看 `server.cpp`（发送端）主逻辑：
- 总体流程：
  1. `initHots();` 初始化源/目的主机信息（把寄件人/收件人信息写到寄件单上）。
  2. 进入循环，提示用户输入消息（`fgets` 读取），如果输入 `quit` 则退出。
  3. 计算消息长度 `len = strlen((const char*)data);`
  4. `MAC mac = encapsulateData(data, len);` 把消息逐层封装（构造好 OCP→UDP→IP→MAC）。
  5. 计算各层长度并得出 `lenMAC`（整帧字节数）。
  6. `frame = malloc(lenMAC); macToByte(mac, lenMAC, frame);` 分配内存并把结构体写成字节数组。
  7. `displayByte("Frame", frame, lenMAC); displayMAC(mac);` 打印帧和结构体内容以供查看（类似核对单据）。
  8. `FILE *fpWrite = fopen("test.txt", "wb"); writeMessage(frame, lenMAC, fpWrite); fclose(fpWrite);` 写到文件。
  9. `free(frame);` 释放内存。
- 逐行解释关键函数调用（补充细节）：
  - `malloc(lenMAC)`：向操作系统请求 `lenMAC` 字节的内存，返回一个指向它的指针。类比：临时准备一个托盘来放整箱数据。
  - `macToByte(...)`：把结构化的 `mac` 转换为连续字节并更新校验。
  - 在写完文件后释放 `frame`，这是良好实践，避免内存泄露（就像用完托盘要收回）。

这是在把包裹包装好并放入邮筒并锁好门的流程。

再看 `client.cpp`（接收端）主逻辑：
- 总体流程：
  1. `initHots();` 初始化本机和目标信息。
  2. 等待用户输入 `y`（表示要取包裹）。
  3. `FILE *fp = fopen("test.txt","rb+");` 打开文件读取。
  4. `short lenMAC = getLenMAC(fp);` 先只读头部来获得整帧长度。
  5. `frame = malloc(lenMAC); readMessage(frame, lenMAC, fp);` 申请内存并把整帧读入内存后关闭文件。
  6. `displayByte("frame", frame, lenMAC);` 打印完整帧（十六进制）。
  7. `checkFCS(frame, lenMAC);` 校验帧的 FCS 是否通过。
  8. `checkSum(frame);` 校验 IP / UDP / OCP 校验和。
  9. `MAC mac = bytesToMAC(frame, Data); displayMAC(mac); decapsulateMAC(mac);` 把字节帧解析成结构体并逐层检查（地址/端口等），最后输出消息接收成功或错误信息。
 10. `free(frame);`
- 逐行关键说明：
  - `getLenMAC` 只看头部确定长度，然后 `readMessage` 读完整帧。这样避免一次读不够或读多了。
  - `checkFCS` 和 `checkSum` 用来判定帧是否在“传输（文件写/读）过程中被损坏或被篡改”。
  - `bytesToMAC` 会做大量 `memcpy` 来解析帧里的每个字段。

这是邮递员从邮筒取出包裹、扫描封条、打开包裹并交付到正确部门的流程。

## 第四步：总结（用大白话串联并指出最核心的 3 个知识点）

- 用大白话再串一次流程（寄快递式）：
  1. 发送端（`server.cpp`）把要寄的东西（字符串消息）装进小盒子（OCP），把小盒子放进中盒（UDP），再放进大盒（IP），外面套上外包装（MAC）。每一层盒子上都贴有标签（地址、端口、长度）并写上校验“签名”（IP 校验和、UDP 校验、OCP 校验、帧尾的 CRC/FCS）。这就是“封装”过程，相当于依次填好所有运单和标签。
  2. 把这个“最终的包”放到一个共享位置（文件 `test.txt`，模拟网线/媒介）。
  3. 接收端（`client.cpp`）从共享位置取走这个包，先扫一遍封条（CRC）看有没有裂开，然后核对每级运单的签名（IP/UDP/OCP 校验和），最后拆开每层盒子，确认确实寄给自己的（MAC、IP、端口），把最里边的东西（消息）拿出来给对应的应用。
  4. 如果在任何一步校验失败，就说明包裹在途中被损坏或不是寄给你的。

- 这段代码里最核心的 3 个知识点（大白话）：
  1. 分层封装与解封装（encapsulation / decapsulation）：数据从应用到传输到网络到链路，每层加上自己的头部，接收端按相反顺序一层层拆开。类比：小盒子套大盒子。
  2. 字节与端序转换（shortToByte / bytesToShort）：真实网络上传输的是“字节序列”，数字要拆成字节或从字节合成数字，必须按一致的顺序（大端/小端）处理。类比：把两位数拆成“十位+个位”再组合。
  3. 校验和与完整性检查（CRC32 / IP checksum / UDP checksum）：网络上会在不同层做签名（不同算法）以检测数据是否被损坏或被篡改。类比：贴封条、在单据上签名并在接收时复核。

附加：常见容易混淆的点（小心台阶）
- 结构体的数组字段（`byte data[256]`）和指针字段（`byte* data`）不同：数组的存储空间就在结构体里；指针只是一个地址，需要保证所指向的内存还在（否则会出现野指针）。
- `memcpy`：它是“逐字节复制”，不会检查目标是否有足够空间，使用时必须保证目标能放下源，否则会内存覆盖。类比：把资料粘贴到表格上，如果表格没有空格就会把旁边的内容挤坏。
- 位运算符(`<<`, `>>`, `&`, `|`, `~`, `^`)：都可以用“积木左移/右移”和“按位体检/合并”来理解，不要被符号吓到。比如 `x << 8` 就是把 `x` 向左移动 8 位（把低位腾出来放高位）。
